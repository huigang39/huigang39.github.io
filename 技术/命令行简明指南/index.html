# 命令行简明指南

# 序言

在我看来，在命令行界面（Command-Line Interface；CLI）输入各种指令是一种高效、优雅、便捷且装 X 的操作方式（当然前提是你得会用），图形化用户界面（Graphical User Interface；GUI）虽然直观，符合人类的日常操作习惯，但是它不够通用 —— 比如说，当一款软件进行大版本升级迭代之后，往往他的 GUI 的展现方式可能就会发生变化，各种功能按钮因此会移位，外观也可能因此发生变化。并且 GUI 的操作方式注定了它没有批量操作的可行性，也就是自动化。例如，当你想要修改一万个文件的名字使它们呈现某种规律（例如按当前的时间命名）的时候，你不得不借助第三方的工具来帮助你完成任务，又例如，当你想批量管理 Kubernetes 集群容器时，难道你要一个一个的去点击按钮操作吗？而在命令行的操作方式里，我们完全可以写一个脚本（事实上它就是一种图灵完备的计算机语言）来完成上述种种操作。除此之外，命令行之所以还存在是因为大部分的服务器都是搭载的 Linux 系统（虽说 Windows 也有 Server 版本，但是用的不多，而且即便是 Windows Server，在服务器上操作时大部分也是使用命令行的方式来操作），并且全都是使用命令行进行操作（废话，谁没事给服务器装上显示器，钱多是吧？），当然你也可以使用 X Server 来给你的客户端的显示器远程转发图形化界面，但是很显然这是没有必要的（除非有特殊情况）。图形化的华丽界面带来的代价就是计算机资源的大量占用，显卡需要耗费大量计算能力的去渲染我们人眼需要看到的画面；而在命令行下就不需要，这也体现出了命令行的低能耗的优点。

综上所述，我们得出命令行至少有三个优点：

  1. 通用性，命令行是通用的，当然，我的意思不是说 Bash Shell 能和 PowerShell 的脚本语言能够运行在彼此的运行环境里（不同 Shell 脚本语言的语法稍有不同），而是说操作逻辑是相通的；况且，现在大多数脚本语言已经实现跨平台了，例如 Microsoft 的 PowerShell 完全可以运行在 Linux 和 Mac OS 上，而不再局限于 Windows，Bash、ZSH 等 Linux Shell 也早都移植到了 Windows 上面。更重要的是，在各种极端条件下，命令行往往是你的救命稻草，比如系统崩溃无法加载图形化程序、对嵌入式设备进行调试，这些场景，你可能会需要使用 SSH 远程连接设备，使用 VIM 编辑文件……
  2. 自动化，命令行是完成各种自动化任务的不二之选，学会了使用命令行（Shell 脚本）就能够更高效的完成任务。
  3. 低能耗，命令行界面并不会占用过多的计算机资源，你可以把有限的资源用在更需要它的地方。

最后，还是想补充一下，命令行程序并不能完全代替图形化软件，诸如视频剪辑播放、图像处理、建模仿真等专业软件还是需要用到图形化界面的，并且可视化的界面更利于软件的推广传播而不局限于专业人员。因此，CLI 和 GUI 并不是站在对立面，而是一种相辅相成的关系。
